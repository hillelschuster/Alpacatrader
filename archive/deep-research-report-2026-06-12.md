## Bottom line

The most important conclusion is not subtle: **current runtime behavior is structurally unsafe and strategically misleading**. The app loop claims to run scanner ŌåÆ enrichment ŌåÆ market data ŌåÆ pipeline ŌåÆ execution, but in the scan path it can discard the enriched `MarketSnapshot` and replace it with a bare `MarketSnapshot(candidate=c)`, which strips bars, VWAP, spread, RVOL, and dollar-volume context before calling `run_pipeline()`. Since entry detection in `run_pipeline()` requires both `att_mult > 0.25` and `bars`, that bug alone can turn the live app loop into a de facto no-entry machine even when fresh market data exists. The existing app-level test for ŌĆ£valid market data reaches entry pathŌĆØ is too weak to catch this because it accepts either `"enter"` **or** `"watch"`. Execution truth is also broken in ways that matter operationally. In the monitor path, `submit_exit()` returns a tuple, but the app stores it in `order` and then immediately accesses `order.order_id`; that will fail once an exit is triggered. Partial exit intent is also lost because the app never forwards `exit_pct`, while the gateway defaults exit submission to `100`. On the Alpaca path, both entry fill confirmation and exit fill confirmation can mark local state as filled/closed even when the real broker check fails or never proves a fill. ## New findings the reports missed

The biggest missed finding is the **app-loop snapshot discard**. In `_scan_and_process()`, if `market_data_fn` is configured and returns a non-`None` snapshot, the code still executes `snapshot = MarketSnapshot(candidate=c)` in the `else` branch, overwriting the returned market data with a blank snapshot. That wipes out bars, VWAP, EMA9, spread, quote age, RVOL, and dollar-volume fields right before the pipeline call. Operationally, this is devastating: the app loop can look ŌĆ£wiredŌĆØ while actually starving its own entry engine. The report did not call this out clearly enough, and it matters more than the once-mode bug because this is the actual continuous runtime path. Severity: **critical**. Evidence: `src/app.py:354-365` behavior visible in the scan-path snippet, plus `run_pipeline()`ŌĆÖs `bars` gate for entries. ## Architecture assessment

The module decomposition is the best part of the project. Scanner, attention, hard filters, classifier, entries, sizing, execution, exits, and state machine are separated cleanly enough that the failure modes are visible rather than buried. That is real value. The problem is that the runtime wiring lies about what is actually live. The codebase presents itself as an attention-first execution system, but the app loop does not actually rank top attention before processing, the classifier is only partially fed, and the position side still relies on local synthetic truths rather than broker/protection truths. ŅłĆciteŅłéturn40view0Ņłéturn29view3Ņłéturn46view0Ņłéturn49view0Ņłü

`run_pipeline()` is overloaded. It is trying to be both a candidate-evaluation engine and a live-position monitor/orchestrator. That is structurally wrong. Candidate flow and position flow have different truths, different data contracts, and different failure semantics. Candidate flow should own discovery, enrichment, attention, hard filters, classification, entry detection, and sizing. Position flow should own broker truth, fill truth, protection truth, stop-order identity, realized/unrealized P&L, and exit policy. Right now those concerns are entangled, which is why the monitor path rebuilds a fake `Candidate` for an open position and loses original entry context. ŅłĆciteŅłéturn29view1Ņłéturn29view2Ņłéturn45view0Ņłü

The data model is too thin for serious execution correctness. `PositionStateModel` carries price, shares, stop, and P&L placeholders, but it does not obviously persist the fields a real exit/protection engine needs: broker entry order id, broker stop order id, entry setup, entry spread, initial risk per share, scale-out history, or fill ledger. Without those fields, the system keeps having to reconstruct truth from partial local state, which is exactly why protection truth, partial-exit truth, and reconciliation truth are weak. ŅłĆciteŅłéturn35view0Ņłéturn35view2Ņłéturn37view3Ņłéturn37view4Ņłü

The code also overclaims around live strategy viability. The spec says ŌĆ£process top-attention candidates onlyŌĆØ and frames the bot as a momentum execution machine, not a rejection engine. But in current runtime reality, the loop is sequential, the classifier is underfed, the app scan path can drop actual market data, hard filters fabricate bid/ask estimates from the candidate price, and the practical effect is still closer to a disguised rejection system than to an execution system. ŅłĆciteŅłéturn49view0Ņłéturn49view2Ņłéturn29view3Ņłéturn28view0Ņłéturn28view1Ņłü

## Risk and safety failure modes

Worst-case failure mode one is **local broker divergence while the bot believes it is safe**. A local entry can become `OPEN` without a real filled broker buy in the Alpaca path. A local exit can become `CLOSED` without a verified filled broker sell. A local position can appear protected because `stop_price` is set, while no live protective order exists. This combination is enough to corrupt every higher-level control: open-position count, open-risk budgeting, missing-protection handling, and whether the symbol should be locked. ŅłĆciteŅłéturn38view5Ņłéturn38view6Ņłéturn38view7Ņłéturn39view0Ņłéturn39view3Ņłéturn38view2Ņłéturn36view7Ņłü

Worst-case failure mode two is **risk-stop bypass by accounting blindness**. The app builds `AccountRiskState` from open positions only. Closed losses drop out. `AccountRiskState` exposes daily P&L and per-symbol loss structures, but the runtime coding pattern means those values are not reliable session ledgers. That creates a path where the bot can lose money, close the losing trade, watch the loss disappear from the gating snapshot, and continue trading despite supposed daily-stop logic. ŅłĆciteŅłéturn27view0Ņłéturn35view2Ņłéturn40view1Ņłü

Worst-case failure mode three is **unprotected restart recovery**. Reconciliation exists, and the spec correctly says broker truth wins after restart. But the startup path only reacts to one action shape (`insert_protect`) and ignores the other reconciliation actions. If a broker-side position is inserted without a stop price into local state, the startup handler explicitly will not call `protect_position()` because it requires `pos.stop_price is not None`. That is how you restart into an open but unprotected state without an immediate hard safety response. ŅłĆciteŅłéturn26view3Ņłéturn37view10Ņłü

Worst-case failure mode four is **false emergency exits during data outages**. The monitor path substitutes `current_price = 0.0` and stale age `999.0` on market-data failure, then allows the exit engine to see what looks like crisis context. In a real degraded-data scenario, the correct design is to distinguish ŌĆ£quote unavailableŌĆØ from ŌĆ£price is zeroŌĆØ and then take a controlled degraded-mode action. Here the degraded mode is entangled with the emergency exit path itself. ŅłĆciteŅłéturn27view2Ņłéturn27view3Ņłéturn42view2Ņłü

If only three things get fixed, fix these.

First, **separate broker truth from local intent**. No position may become locally `OPEN` without a real fill-confirmed broker qty, and no position may become locally `CLOSED` without a real exit fill confirmation or an explicitly labeled synthetic-paper mode path that cannot leak into Alpaca mode.
